Client ↔ server over HTTP

Same idea as your sketch: request out, response back — with status codes in the response.

Status codes

2xx — successful response

Client browser, app, Postman…
HTTP request
e.g. GET /api/cat
Server Express, API…
Client
HTTP response
body + headers
Server
200 OK · family 2xx = success

The status code lives on the response (e.g. 200). The client uses it to know the request succeeded.

Redirects

3xx — redirect (client should go elsewhere)

Client follows Location
HTTP request
e.g. GET /old-path
Server “moved” or “see other”
Client
HTTP response
Location: https://…/signin
Server
302 Found · family 3xx = redirect

The server answers with a 3xx and often a Location header. The browser then issues a new request to that URL (your “tidy URL” flow).
